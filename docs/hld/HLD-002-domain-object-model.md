# HLD-002: Domain Object Model

## Status

Draft

## Purpose

This document defines the core domain objects for the **EDD stack**.

The purpose of this HLD is to give implementation agents a shared vocabulary and prevent important concepts from being collapsed into vague or overloaded objects.

The EDD stack is not only a run tracker, eval dashboard, LangGraph playground, or Langfuse wrapper. It is a workflow system for agent design. The domain model must preserve that workflow.

The core lifecycle is:

```text
agent idea
  → target
  → rules
  → eval contract
  → information requirements
  → tool requirements
  → tool feasibility
  → graph design
  → v0
  → trace evidence
  → failure packets
  → bounded fix
  → v1
  → comparison
  → gates
  → promotion
  → operation
```

Each major step should have a corresponding domain object.

See [HLD-001: Product Intent and System Boundaries](HLD-001-product-intent-and-system-boundaries.md) for system roles and repository boundaries.

---

## Design Principles

### 1. Preserve conceptual separation

Do not collapse distinct concepts into generic objects.

For example:

```text
ToolRequirement != ToolImplementation != ToolBinding != ToolFeasibilityReview
```

These are separate because they answer different questions:

| Concept | Question |
|---|---|
| **ToolRequirement** | What kind of tool is needed? |
| **ToolImplementation** | What concrete implementation exists? |
| **ToolBinding** | Which implementation is active for a graph node or environment? |
| **ToolFeasibilityReview** | Is this realistic for demo, internal, or production use? |

### 2. Design intent is first-class

Agent behavior should not be defined only by code, prompts, or traces.

The platform must model:

- AgentTarget
- BehaviorRule
- EvalContract
- InformationRequirement
- ToolRequirement
- GraphDesign

These objects define what the agent is supposed to do before judging whether it succeeded.

### 3. Runtime evidence is linked, not owned

The platform owns the meaning of a run, but Langfuse owns detailed trace evidence.

The platform should store trace references and metadata through `TraceLink`, not duplicate all trace data.

### 4. Version decisions are explicit

Promotion should be modeled as a decision record.

Do not infer promotion only from “latest version” or “best score.”

The platform should record:

- what version was promoted
- why it was promoted
- what gates passed or failed
- what risks were accepted
- what environment it applies to

### 5. Generated artifacts remain reviewable

Many objects may be generated by the system, but users should be able to review, edit, approve, or reject them.

Examples:

- BehaviorRule
- EvalContract
- InformationRequirement
- ToolRequirement
- GraphDesign
- FixPlan

Generated does not mean canonical until accepted or saved.

## Core Objects

### Agent

#### Purpose

Represents a named AI agent being designed, evaluated, versioned, and potentially operated.

#### Examples

- Customer Escalation Triage Agent
- Strategic Deployment Copilot
- Daily Work Prioritization Agent

#### Key Fields

```yaml
Agent:
  id: string
  name: string
  description: string
  status: draft | active | deprecated
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- Agent has many AgentTargets.
- Agent has many AgentVersions.
- Agent has many ExperimentRuns.
- Agent has many PromotionRecords.

---

### AgentTarget

#### Purpose

Defines what the agent is supposed to do.

This is the starting point for evaluation-driven design.

The target should describe purpose, intended users, goals, non-goals, risk areas, and expected behavior.

#### Key Fields

```yaml
AgentTarget:
  id: string
  agent_id: string
  version: string
  name: string
  purpose: string
  intended_users: list[string]
  primary_goals: list[string]
  non_goals: list[string]
  risk_areas: list[string]
  expected_output_style: string
  status: draft | active | archived
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- AgentTarget belongs to Agent.
- AgentTarget has many BehaviorRules.
- AgentTarget has many EvalContracts.
- AgentTarget has many InformationRequirements.
- AgentTarget is referenced by AgentVersion and ExperimentRun.

#### Notes

The active target defines what “good” means for an agent at a point in time.

Changing the target is a meaningful design change and should be versioned.

---

### BehaviorRule

#### Purpose

Defines a specific expectation for agent behavior.

Behavior rules convert target intent into explicit, evaluable design constraints.

#### Example

```yaml
BehaviorRule:
  id: separate_facts_from_hypotheses
  description: >
    The agent must distinguish confirmed facts from likely causes and unknowns.
  severity: critical
```

#### Key Fields

```yaml
BehaviorRule:
  id: string
  agent_target_id: string
  name: string
  description: string
  severity: critical | high | medium | low
  category: discovery | grounding | evidence | tool_use | planning | safety | communication | operations
  status: draft | active | disabled | archived
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- BehaviorRule belongs to AgentTarget.
- BehaviorRule may be used by Metrics.
- BehaviorRule may require InformationRequirements.
- BehaviorRule may be supported by GraphNodes.
- BehaviorRule may be violated by FailurePackets.

#### Notes

Rules should not be passive documentation.

They should connect to eval metrics, graph nodes, failures, gates, and fixes.

---

### EvalContract

#### Purpose

Defines how agent behavior will be evaluated.

An eval contract includes metrics, rubrics, scoring scales, gates, and scenario bindings.

#### Key Fields

```yaml
EvalContract:
  id: string
  agent_target_id: string
  version: string
  name: string
  description: string
  metrics: list[Metric]
  gates: list[Gate]
  scenario_set_ids: list[string]
  status: draft | active | archived
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- EvalContract belongs to AgentTarget.
- EvalContract contains Metrics.
- EvalContract contains Gates.
- EvalContract references ScenarioSets.
- EvalContract is referenced by ExperimentRuns and Comparisons.

#### Notes

Comparisons between v0 and v1 should normally use the same eval contract.

If the eval contract changes, the change must be explicit and versioned.

---

### Metric

#### Purpose

Defines a measurable evaluation dimension.

#### Example

```yaml
Metric:
  id: diagnostic_grounding
  scale: 0-5
  rules:
    - evidence_first_diagnosis
    - separate_facts_from_hypotheses
```

#### Key Fields

```yaml
Metric:
  id: string
  eval_contract_id: string
  name: string
  description: string
  scale_min: number
  scale_max: number
  rubric: string
  behavior_rule_ids: list[string]
  weight: number
```

#### Relationships

- Metric belongs to EvalContract.
- Metric references BehaviorRules.
- Metric produces Scores.
- Metric may be used by Gates.

---

### Gate

#### Purpose

Defines a pass/fail/warning condition used to determine behavior readiness, tool readiness, operational readiness, or promotion readiness.

#### Key Fields

```yaml
Gate:
  id: string
  eval_contract_id: string
  name: string
  description: string
  type: hard | warning | informational
  category: behavior | tool_readiness | operational_safety | regression | overfitting | cost | latency
  condition: string
  severity: critical | high | medium | low
```

#### Relationships

- Gate belongs to EvalContract.
- Gate may reference Metrics.
- Gate produces GateResults.
- Gate may block PromotionRecords.

#### Notes

A version can pass behavior gates and still fail production readiness gates.

---

### Scenario

#### Purpose

Represents one test case, customer situation, or operational situation used to evaluate an agent.

#### Key Fields

```yaml
Scenario:
  id: string
  name: string
  description: string
  user_prompt: string
  context: object
  expected_behavior: list[string]
  hidden_expectations: list[string]
  tags: list[string]
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- Scenario may belong to one or more ScenarioSets.
- Scenario is used by ExperimentRuns.
- Scenario may produce FailurePackets.

---

### ScenarioSet

#### Purpose

A collection of scenarios used for a run, regression suite, overfitting check, or promotion gate.

#### Key Fields

```yaml
ScenarioSet:
  id: string
  name: string
  description: string
  scenario_ids: list[string]
  purpose: baseline | regression | variant | promotion | smoke | demo
```

#### Relationships

- ScenarioSet contains Scenarios.
- ScenarioSet is referenced by EvalContracts.
- ScenarioSet is used by ExperimentRuns.

---

### InformationRequirement

#### Purpose

Defines information the agent needs in order to satisfy behavior rules.

Information requirements should be generated before tool requirements.

#### Example

```yaml
InformationRequirement:
  id: trace_evidence
  required_for_rules:
    - evidence_first_diagnosis
    - separate_facts_from_hypotheses
  description: >
    Recent traces, model calls, tool calls, latency, failed spans, and error patterns.
```

#### Key Fields

```yaml
InformationRequirement:
  id: string
  agent_target_id: string
  name: string
  description: string
  behavior_rule_ids: list[string]
  required: boolean
  sensitivity: public | internal | confidential | regulated
  status: draft | active | archived
```

#### Relationships

- InformationRequirement belongs to AgentTarget.
- InformationRequirement references BehaviorRules.
- InformationRequirement may be satisfied by ToolRequirements.
- InformationRequirement may be used by GraphNodes.

#### Notes

This object answers: **What information does the agent need?**

It does not answer: **Which tool exists?**

---

### ToolRequirement

#### Purpose

Defines a type of tool or connector needed to satisfy an information requirement or support an action.

A tool requirement is not the same as an implemented tool.

#### Example

```yaml
ToolRequirement:
  id: trace_evidence_source
  information_requirement_id: trace_evidence
  suggested_tool_name: fetch_trace_summary
  access_mode: read_only
```

#### Key Fields

```yaml
ToolRequirement:
  id: string
  information_requirement_id: string
  suggested_tool_name: string
  purpose: string
  access_mode: read_only | approval_gated_write | controlled_automation
  required_maturity_level: missing | mock | local | read_only_live | approval_gated_action | controlled_automation
  required_for_demo: boolean
  required_for_production: boolean
  status: draft | active | archived
```

#### Relationships

- ToolRequirement satisfies an InformationRequirement.
- ToolRequirement may have many ToolImplementations.
- ToolRequirement may be referenced by GraphNodes.
- ToolRequirement may be evaluated by ToolFeasibilityReviews.

#### Notes

This object answers: **What kind of tool is needed?**

It does not answer:

- Does the tool exist?
- Which implementation is active?
- Is it production ready?

---

### ToolImplementation

#### Purpose

Represents a concrete implementation of a tool requirement.

Examples:

- mock JSON fixture
- local artifact reader
- Langfuse API connector
- GitHub API connector
- Jira API connector
- approval-gated Slack message tool

#### Key Fields

```yaml
ToolImplementation:
  id: string
  tool_requirement_id: string
  name: string
  mode: missing | mock | local | read_only_live | approval_gated_action | controlled_automation
  status: not_implemented | available | failing | disabled | deprecated
  backing_source: string
  description: string
  config_schema: object
  output_schema: object
```

#### Relationships

- ToolImplementation implements ToolRequirement.
- ToolImplementation may be selected by ToolBinding.
- ToolImplementation may be used during ExperimentRuns or OperationalRuns.

#### Notes

Mock implementations are valid for demos and evaluation loops, but they are not production-ready.

---

### ToolBinding

#### Purpose

Defines which tool implementation is active for a graph node, agent version, environment, or run.

#### Key Fields

```yaml
ToolBinding:
  id: string
  tool_requirement_id: string
  tool_implementation_id: string
  graph_node_id: string
  agent_version_id: string
  environment: local_demo | test | internal | production
  active: boolean
```

#### Relationships

- ToolBinding connects ToolRequirement, ToolImplementation, GraphNode, and AgentVersion.
- ToolBinding is referenced by ExperimentRuns and OperationalRuns.

#### Notes

This object answers: **For this version/environment/node, which implementation is being used?**

Example:

```yaml
ToolBinding:
  graph_node_id: collect_trace_evidence
  tool_requirement_id: trace_evidence_source
  tool_implementation_id: fetch_trace_summary_mock
  environment: local_demo
```

---

### ToolFeasibilityReview

#### Purpose

Assesses whether a tool requirement and its implementations are feasible for demo, internal, or production use.

#### Key Fields

```yaml
ToolFeasibilityReview:
  id: string
  tool_requirement_id: string
  status: feasible | feasible_for_demo | blocked | needs_review
  demo_ready: boolean
  production_ready: boolean
  risks: list[string]
  blockers: list[string]
  recommended_next_step: string
  reviewed_by: string
  reviewed_at: datetime
```

#### Relationships

- ToolFeasibilityReview belongs to ToolRequirement.
- ToolFeasibilityReview may influence GateResults and PromotionRecords.

#### Notes

This object keeps the platform honest.

The system must not imply production readiness when required tools are mock-only.

---

### GraphDesign

#### Purpose

Defines the intended graph architecture for an agent version or target.

Graph design should explain how rules, information requirements, tool requirements, and failure packets influence graph structure.

#### Key Fields

```yaml
GraphDesign:
  id: string
  agent_target_id: string
  eval_contract_id: string
  version: string
  name: string
  description: string
  graph_node_ids: list[string]
  status: draft | active | archived
```

#### Relationships

- GraphDesign belongs to AgentTarget.
- GraphDesign references EvalContract.
- GraphDesign contains GraphNodes.
- GraphDesign may be implemented by AgentVersion.

---

### GraphNode

#### Purpose

Represents a logical node in a graph design.

#### Example

```yaml
GraphNode:
  id: separate_facts_hypotheses_unknowns
  purpose: >
    Prevent unsupported diagnosis by separating confirmed facts, plausible hypotheses,
    and unknowns before recommendations are generated.
```

#### Key Fields

```yaml
GraphNode:
  id: string
  graph_design_id: string
  name: string
  purpose: string
  behavior_rule_ids: list[string]
  information_requirement_ids: list[string]
  tool_requirement_ids: list[string]
  prompt_ids: list[string]
  reads_state: list[string]
  writes_state: list[string]
```

#### Relationships

- GraphNode belongs to GraphDesign.
- GraphNode may support BehaviorRules.
- GraphNode may use InformationRequirements.
- GraphNode may require ToolRequirements.
- GraphNode may use Prompts.
- GraphNode may be linked to FailurePackets or FixPlans.

#### Notes

Every significant graph node should have a reason to exist.

Acceptable reasons include:

- behavior rule
- information requirement
- tool requirement
- failure packet
- fix plan
- operational safety requirement

---

### Prompt

#### Purpose

Represents a prompt artifact used by a graph node or agent version.

#### Key Fields

```yaml
Prompt:
  id: string
  name: string
  graph_node_id: string
  agent_version_id: string
  file_path: string
  content_hash: string
  purpose: string
  behavior_rule_ids: list[string]
```

#### Relationships

- Prompt may belong to GraphNode.
- Prompt may belong to AgentVersion.
- Prompt may support BehaviorRules.
- Prompt may be changed by FixPlans.

#### Notes

Prompt files may live in the lab repo, but the platform should be able to reference them as artifacts.

---

### AgentVersion

#### Purpose

Represents a concrete version of an agent implementation.

#### Examples

- v0-baseline
- v1-evidence-triage-graph
- v2-live-trace-reader

#### Key Fields

```yaml
AgentVersion:
  id: string
  agent_id: string
  source_version_id: string | null
  target_id: string
  eval_contract_id: string
  graph_design_id: string
  version_label: string
  implementation_summary: string
  status: draft | candidate | promoted | rejected | deprecated
  tool_mode_summary: string
  created_at: datetime
  updated_at: datetime
```

#### Relationships

- AgentVersion belongs to Agent.
- AgentVersion references AgentTarget.
- AgentVersion references EvalContract.
- AgentVersion may implement GraphDesign.
- AgentVersion may have ToolBindings.
- AgentVersion has ExperimentRuns.
- AgentVersion may be compared with other AgentVersions.
- AgentVersion may be promoted through PromotionRecord.

#### Notes

A version should not be accepted as “better” only because it has a higher score. The improvement should be tied to resolved failures and gate results.

---

### ExperimentRun

#### Purpose

Represents an evaluation or test run of an agent version against scenarios.

#### Key Fields

```yaml
ExperimentRun:
  id: string
  agent_id: string
  agent_version_id: string
  target_id: string
  eval_contract_id: string
  scenario_set_id: string
  environment: local_demo | test | internal | production
  tool_mode_summary: string
  status: queued | running | completed | failed | partial
  started_at: datetime
  completed_at: datetime
```

#### Relationships

- ExperimentRun belongs to AgentVersion.
- ExperimentRun references AgentTarget and EvalContract.
- ExperimentRun uses ScenarioSet.
- ExperimentRun has Scores.
- ExperimentRun has TraceLinks.
- ExperimentRun may produce FailurePackets.
- ExperimentRun may produce GateResults.

---

### OperationalRun

#### Purpose

Represents real use of a promoted agent, as opposed to an eval run.

Examples:

- Run the Customer Escalation Triage Agent on an active customer escalation.
- Run the Daily Work Prioritization Agent for the current workday.

#### Key Fields

```yaml
OperationalRun:
  id: string
  agent_id: string
  agent_version_id: string
  promotion_record_id: string
  target_id: string
  eval_contract_id: string
  environment: internal | production
  tool_mode_summary: string
  user_input: string
  output_summary: string
  status: completed | failed | needs_review
  created_at: datetime
```

#### Relationships

- OperationalRun belongs to promoted AgentVersion.
- OperationalRun references PromotionRecord.
- OperationalRun may have TraceLinks.
- OperationalRun may produce action recommendations.
- OperationalRun may be reviewed or evaluated after the fact.

#### Notes

Operational runs must preserve traceability to the target, eval contract, version, and tool bindings.

---

### TraceLink

#### Purpose

References trace evidence stored outside the platform, usually in Langfuse.

#### Key Fields

```yaml
TraceLink:
  id: string
  provider: langfuse
  external_trace_id: string
  external_url: string
  experiment_run_id: string | null
  operational_run_id: string | null
  scenario_id: string | null
  agent_version_id: string
  metadata: object
```

#### Relationships

- TraceLink belongs to ExperimentRun or OperationalRun.
- TraceLink may be referenced by FailurePackets.
- TraceLink may be referenced by GateResults.

#### Notes

The platform should not duplicate all trace contents. It should store enough metadata and links to connect platform decisions to Langfuse evidence.

---

### Score

#### Purpose

Represents a metric score for a run, scenario, trace, or version.

#### Key Fields

```yaml
Score:
  id: string
  metric_id: string
  experiment_run_id: string
  scenario_id: string | null
  trace_link_id: string | null
  value: number
  scale_min: number
  scale_max: number
  explanation: string
  evaluator: string
```

#### Relationships

- Score belongs to ExperimentRun.
- Score references Metric.
- Score may reference Scenario.
- Score may reference TraceLink.
- Score may be used by GateResult.

---

### FailurePacket

#### Purpose

Connects a failed rule to observed behavior, trace evidence, suspected cause, and recommended fix.

#### Key Fields

```yaml
FailurePacket:
  id: string
  experiment_run_id: string
  agent_version_id: string
  scenario_id: string
  failed_behavior_rule_id: string
  severity: critical | high | medium | low
  observed_behavior: string
  expected_behavior: string
  suspected_cause: string
  recommended_fix: string
  trace_link_ids: list[string]
  status: open | addressed | resolved | accepted_risk
```

#### Relationships

- FailurePacket belongs to ExperimentRun.
- FailurePacket references AgentVersion.
- FailurePacket references BehaviorRule.
- FailurePacket references TraceLinks.
- FailurePacket may be addressed by FixPlan.

#### Notes

Failure packets are one of the most important objects in the EDD stack.

They turn eval failure into design action.

---

### FixPlan

#### Purpose

Defines a bounded change proposal derived from failure packets.

#### Key Fields

```yaml
FixPlan:
  id: string
  source_version_id: string
  target_version_label: string
  failure_packet_ids: list[string]
  behavior_rule_ids_addressed: list[string]
  graph_changes: list[string]
  prompt_changes: list[string]
  tool_changes: list[string]
  non_goals: list[string]
  regression_risks: list[string]
  status: draft | approved | implemented | rejected
```

#### Relationships

- FixPlan addresses FailurePackets.
- FixPlan creates or informs a new AgentVersion.
- FixPlan may modify GraphDesign, Prompts, or ToolBindings.

#### Notes

A fix plan should be bounded. It should not silently redesign everything.

---

### Comparison

#### Purpose

Compares two or more agent versions against the same target and eval contract.

#### Key Fields

```yaml
Comparison:
  id: string
  baseline_version_id: string
  candidate_version_id: string
  target_id: string
  eval_contract_id: string
  scenario_set_id: string
  score_deltas: object
  resolved_failure_packet_ids: list[string]
  new_failure_packet_ids: list[string]
  regression_warnings: list[string]
  summary: string
```

#### Relationships

- Comparison references AgentVersions.
- Comparison references EvalContract.
- Comparison may reference FailurePackets.
- Comparison may produce GateResults.
- Comparison may inform PromotionRecord.

#### Notes

A comparison should explain why a version improved or regressed. It should not only show a score delta.

---

### GateResult

#### Purpose

Represents the result of applying gates to a version, run, or comparison.

#### Key Fields

```yaml
GateResult:
  id: string
  eval_contract_id: string
  agent_version_id: string
  experiment_run_id: string | null
  comparison_id: string | null
  gate_id: string
  status: pass | fail | warning | not_applicable
  evidence: string
  trace_link_ids: list[string]
  blockers: list[string]
```

#### Relationships

- GateResult references Gate.
- GateResult references AgentVersion.
- GateResult may reference ExperimentRun or Comparison.
- GateResult may block PromotionRecord.

---

### PromotionRecord

#### Purpose

Records a decision to promote, reject, deprecate, or accept a version with risk.

#### Key Fields

```yaml
PromotionRecord:
  id: string
  agent_id: string
  agent_version_id: string
  previous_version_id: string | null
  decision: promoted_for_demo | promoted_for_internal_use | promoted_for_production_assistive_use | promoted_for_controlled_automation | rejected | deprecated | accepted_with_risk
  rationale: string
  gate_result_ids: list[string]
  accepted_risks: list[string]
  production_status: ready | blocked | not_applicable
  decided_by: string
  decided_at: datetime
```

#### Relationships

- PromotionRecord belongs to Agent.
- PromotionRecord references AgentVersion.
- PromotionRecord references GateResults.
- PromotionRecord may authorize OperationalRuns.

#### Notes

Promotion is an explicit decision. It should not be inferred only from score.

---

### ReadinessStatus

#### Purpose

Summarizes readiness across behavior, tools, operations, and promotion.

#### Key Fields

```yaml
ReadinessStatus:
  id: string
  agent_version_id: string
  behavior_status: pass | fail | warning | not_run
  tool_status: missing | mock_only | local | read_only_live | approval_gated | production_ready
  operational_status: not_ready | demo_ready | internal_ready | production_assistive_ready | controlled_automation_ready
  promotion_status: draft | promoted_for_demo | promoted_for_internal_use | promoted_for_production_assistive_use | promoted_for_controlled_automation | rejected | deprecated
  blockers: list[string]
```

#### Relationships

- ReadinessStatus references AgentVersion.
- ReadinessStatus is informed by GateResults and ToolFeasibilityReviews.

## Object Creation Flow

The intended object creation flow is:

```text
Agent
  ↓
AgentTarget
  ↓
BehaviorRules
  ↓
EvalContract
  ↓
InformationRequirements
  ↓
ToolRequirements
  ↓
ToolImplementations
  ↓
ToolBindings
  ↓
ToolFeasibilityReviews
  ↓
GraphDesign
  ↓
AgentVersion v0
  ↓
ExperimentRun
  ↓
TraceLinks + Scores
  ↓
FailurePackets
  ↓
FixPlan
  ↓
AgentVersion v1
  ↓
Comparison
  ↓
GateResults
  ↓
PromotionRecord
  ↓
OperationalRuns
```

Not every object must be fully implemented in the first milestone, but the architecture should not contradict this flow.

## Minimal MVP Object Set

The first implementation does not need every object fully persisted.

However, the MVP should preserve the conceptual model.

Recommended MVP objects:

```text
Agent
AgentTarget
BehaviorRule
EvalContract
InformationRequirement
ToolRequirement
ToolFeasibilityReview
GraphDesign
AgentVersion
ExperimentRun
TraceLink
Score
FailurePacket
FixPlan
Comparison
GateResult
PromotionRecord
```

Tool implementations and bindings can initially be represented as YAML artifacts if needed, but they should still be modeled separately.

---

## Anti-Patterns

### Anti-pattern: Generic Tool object

Do not model everything as a single `Tool` object.

This loses the difference between need, implementation, binding, and feasibility.

Use:

- ToolRequirement
- ToolImplementation
- ToolBinding
- ToolFeasibilityReview

### Anti-pattern: Eval-only design

Do not model only Run, Score, and Trace.

This would turn the platform into an eval dashboard instead of an EDD system.

### Anti-pattern: Latest version wins

Do not treat the latest version as the active version automatically.

Use `PromotionRecord`.

### Anti-pattern: Langfuse as canonical workflow state

Do not store target, rules, gates, or promotion only as Langfuse metadata.

Langfuse evidence should link back to platform objects.

### Anti-pattern: Silent eval contract mutation

Do not change metrics, rubrics, or gates between v0 and v1 without creating a new EvalContract version.

### Anti-pattern: Production readiness from mock data

Do not mark a version production-ready when required tools are mock-only.

---

## Acceptance Criteria for This HLD

Implementation work is aligned with this domain model when:

1. AgentTarget, BehaviorRule, and EvalContract are first-class concepts.
2. InformationRequirement exists separately from ToolRequirement.
3. ToolRequirement, ToolImplementation, ToolBinding, and ToolFeasibilityReview are not collapsed.
4. GraphDesign can map nodes back to rules, information requirements, and tool requirements.
5. AgentVersion references target, eval contract, and graph design.
6. ExperimentRun records version, target, eval contract, scenario set, and tool mode.
7. TraceLink stores external trace references without making Langfuse the canonical workflow store.
8. FailurePacket connects failed rule, observed behavior, trace evidence, and recommended fix.
9. FixPlan is derived from failure packets and creates bounded changes.
10. Comparison explains score deltas, resolved failures, new failures, and regressions.
11. GateResult separates behavior gates from readiness gates.
12. PromotionRecord records explicit decision and rationale.
13. ReadinessStatus distinguishes behavior quality, tool readiness, operational readiness, and promotion status.

## Summary

The EDD stack requires a domain model that preserves the design loop.

The most important idea is that the platform is not just tracking runs.

It is tracking why an agent exists, what good behavior means, what information the agent needs, what tools are required, whether those tools exist, how the graph is shaped, where versions fail, what fixes are proposed, and why a version is promoted or rejected.

The object model should make the following story possible:

```text
v0 failed because it overclaimed a root cause.
That violated the rule separate_facts_from_hypotheses.
The failure packet linked the failed rule to trace evidence.
The fix plan added evidence normalization and facts/hypotheses separation.
v1 passed behavior gates.
v1 was promoted for demo.
Production readiness remained blocked because trace and tool-health tools were mock-only.
```

If the implementation can represent that story cleanly, the domain model is doing its job.
